## Day 0 — HTML extraction patterns in real Outlook emails

Outlook emails generated by Word have predictable table structure but include
forward-chain duplication. Smoke test against 10 real samples revealed:

1. Forward chains preserve table HTML. Email body often contains the
   bank-credit table TWICE (the authoritative one + a duplicate from the
   forwarded original). Agent must take the FIRST matching table, not all.

2. Multiple bank-credit table formats exist:
   - NEFT format: "Tran Date / Particulars / Credit Amount" (most common)
   - Cheque/IFT format: "Co Code / Collection Date / Chq No & Bank GL /
     Particulars / Collection Amt" (1 sample)
   The triage tool needs to handle both via header keyword matching.

3. Non-remittance emails generate many tables (1 sample had 14 tables) but
   none match expected header patterns. Triage logic must be
   header-keyword-driven, not table-count-driven.

## Day 0 — Triage is rule-based, not LLM-based

Initial design instinct was "LLM classifies email kind." Real data
revealed that classification is deterministic given good table-finding
helpers (bank-credit table present? allocation table present? customer-only
table present?). LLM adds no value at this stage.

This refines the principle from Week 3 (the "wrong stage" lesson):
LLM reasoning earns its place at column-name interpretation and reconciliation
— stages where deterministic rules fail. Triage isn't one of those stages.


## Day 1 — Rule-based triage instead of LLM-based

Initial design instinct was "LLM classifies email kind." Real data revealed
that classification is fully deterministic given good table-finding helpers:
1. Does the body have a bank-credit table?
2. Does it have an allocation table?
3. Does it have a customer-only table?
4. Does it have an 'On A/C - <number>' reference?

Triage = these four boolean signals + hasAttachments → email_kind.

LLM adds zero value here. Adding it would have introduced non-determinism
on what should be a clean rules problem and slowed each classification by
~2 seconds.

### Generalization
This continues the principle from Week 3 (the "wrong stage" lesson):
LLM reasoning earns its place at column-name interpretation (Day 3),
narrative parsing across formats (Day 2), and reconciliation (Day 4).
Triage isn't one of those stages.

### Where the design earns LLM calls
Days 2-3 will add LLM-based nodes to:
- Parse NEFT/RTGS/IMPS/UPI/IFT narratives across the 6 payment modes
- Map varying column header names ("Customer Code" vs "Cust.No" vs "Co Code")
  to canonical schema fields
- Reconcile total bank credits vs total net allocations and explain diffs

Those are tasks where rules fail because of variability. Triage isn't.


## Day 1 — Forward-chain table duplication in real Outlook emails

Real Outlook emails preserve quoted text in forward chains, including
table HTML. Smoke test revealed multiple samples with the bank-credit
table appearing twice — the authoritative one written by the analyst,
plus a duplicate from the original quoted email.

Approach: each table-finder (find_bank_credit_table, find_allocation_table,
find_customer_only_table) returns the FIRST matching table. Since the
authoritative table is at the top of the body and the quoted forward
chain comes after, this naturally selects the right one.

Tested across all 10 samples — 10/10 evals passing.

### What this teaches
Real-world data has structure that synthetic test data lacks. The
"forward chain duplicates the table" pattern wouldn't appear in synthetic
samples; only by running against real client emails did it surface.

Discipline: smoke-test against real data BEFORE writing the agent layer.
Day 0's smoke test (just printing table counts) found this issue while
fixing it required only "use the first table, ignore later duplicates."
Discovering the same issue mid-Day-3 would have wasted hours.
